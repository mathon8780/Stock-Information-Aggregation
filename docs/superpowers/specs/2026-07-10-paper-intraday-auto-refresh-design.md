# 模拟交易分钟 K 自动刷新设计

## 目标

从 Transaction 下单面板把股票加入账户自选股后，后端自动采集该股票的分钟 K；采集完成后，当前账户页面无需手动点击即可显示最新数据。连续添加多只股票和页面暂时切到后台时也不能丢失最终刷新。

## 已确认根因

- 后端 SSE 使用 `event: <type>` 自定义事件名，前端仅设置 `EventSource.onmessage`，因此业务事件没有进入前端回调。
- 自选股同步使用非阻塞全局锁。多个后台任务并发时，未取得锁的任务直接返回 `skipped`，数据库任务仍保持 `pending`，只能等待下次服务启动处理。
- 页面恢复可见时只刷新账户数据，没有刷新当前股票的分钟 K；在后台期间忽略的事件无法补偿。

## 设计

1. SSE 输出改为默认 message 帧，仅输出 `data:`；事件类型继续保留在 JSON 的 `type` 字段。现有前端统一监听器据此过滤事件，无需逐页注册自定义事件。
2. 自选股同步任务改为等待同一把锁后串行执行。取得锁后重新读取任务状态，已经成功或失败的重复调用直接返回现有状态，待处理任务继续采集。
3. `visibility.resumed` 视为分钟 K 刷新事件，页面重新可见时读取当前自选股已经落库的数据。
4. 不增加定时调用外部行情源的轮询。加入自选股时仍只排队一次完整采集，前端在完成事件或恢复可见时读取本地数据库。

## 验证

- 后端测试覆盖 SSE 默认 message 帧和同步任务串行/重复调用行为。
- 前端运行时测试覆盖页面恢复可见后需要刷新分钟 K。
- 集中运行后端测试、前端生产构建和一次真实页面流程：加入无分钟 K 的股票，先显示空状态，后台任务完成后自动出现分钟 K 时间与图表。
- 测试结束只删除本次创建的临时账户，不清理现有用户账户或行情数据。
